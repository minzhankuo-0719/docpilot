## Stage 1 — `doc_preprocessor` Library (v1)

**Date:** 2026-05-07

### Process

1. **Source assets.** The developer requested that the raw PDF and PPTX (sourced from `CLAUDE.md § 4`) be committed directly into `data/raw/` so tests would be reproducible.

2. **Module decomposition.** Four modules with single-responsibility boundaries:
   - `pdf.py` — page-level extraction via PyMuPDF (`fitz`), returning a list of `ParsedPage` dataclasses.
   - `pptx.py` — slide-level extraction via python-pptx, returning a list of `ParsedSlide` dataclasses.
   - `cleaner.py` — pure `clean_text()` handling soft-hyphen line breaks, control characters, and Unicode normalization.
   - `chunker.py` — word-count chunking (default 400 words, 50-word overlap), emitting `Chunk` dataclasses with deterministic `chunk_id`.
   - `__init__.py` — re-exports the public API.

3. **Test suite.** 22 tests split into unit (pure logic on `cleaner` / `chunker`) and integration (conditional skip for `pdf` / `pptx` if source files are absent), so CI passes even without data files present.

4. **Import bug fix.** Initial tests used `from packages.doc_preprocessor import ...` and hit `ModuleNotFoundError` because hatchling packages `packages/doc_preprocessor` as `doc_preprocessor` (without the prefix). Imports were rewritten and all tests passed.

### Key Decisions

| Decision | Rationale |
|---|---|
| Word-count chunking (not token-count) | Avoids a tokenizer dependency; BM25-friendly. |
| `chunk_id = sha256(doc_id:page:idx)[:16]` | Deterministic — identical inputs always yield the same ID, enabling dedup. |
| Commit source files into `data/raw/` | Makes tests reproducible; the source material is a public paper. |

### Verification

- `uv run pytest tests/test_preprocessor.py -v` → 22 passed ✓
- PDF parse: `transformer.pdf` — 15 pages, every page yields text ✓
- PPTX parse: `transformer_presentation.pptx` — all slides yield text ✓

## Stage 1.5 — `doc_preprocessor` v2 (Semantic-Aware Refactor)

**Date:** 2026-05-07

### Problems Reported

After running `scripts/demo_pipeline.py`, the developer inspected the output and surfaced four quality issues:

1. **Cleaner flattens paragraph structure.** `re.sub(r"\n+", " ", text)` collapses every newline — including paragraph breaks — into spaces, destroying paragraph semantics.
2. **Chunker is purely word-counted.** It ignores paragraph and sentence boundaries, slicing arbitrarily at 400 words.
3. **Each page produces only 1–3 chunks, each oversized.** The transformer paper has 600–1000 words per page; at 400 words/chunk this yields 1–3 chunks per page — too coarse for retrieval.
4. **Figure/Table captions bleed into body text.** The PDF parser using `get_text("text")` returns reading-order strings, so captions are concatenated with surrounding paragraphs.

### Design

Claude Code mapped each problem to a specific file/line, proposed a fix matrix, and confirmed direction with the developer before implementing.

| Stage | Fix |
|---|---|
| **parse** | PDF switched to `get_text("blocks")` for paragraph-level blocks. Introduced `Block` dataclass (`text` / `block_type` / `bbox`). Caption regex: `^(Figure\|Fig\.?\|Table\|圖\|表)\s*\d+[:.：。]`. `ParsedPage` / `ParsedSlide` gained `blocks: list[Block]`. |
| **clean** | `clean_text` became paragraph-aware: split on `\n\n+`, clean each paragraph, rejoin with `\n\n`. Added `clean_block_text` for single blocks. |
| **chunk** | New `chunk_blocks(blocks, ...)`: captions/headings emit as standalone chunks (with `block_type` metadata); paragraphs greedily pack up to 220 words; oversized paragraphs split at sentence boundaries `[.!?。！？]`; body chunks stitched together with sentence-level overlap; a `has_fresh` flag prevents pure-overlap flushes. |
| **demo_pipeline** | Visualization for parse/clean now shows block lists with `block_type` badges, making caption isolation visible. |

### API Compatibility

- `parse_pdf` / `parse_pptx` / `clean_text` / `chunk_text` names retained — no caller changes required.
- `chunk_text(chunk_size=, overlap=)` semantics: `overlap` switched from word-count to sentence-count (default 1).

### Verification

- `uv run pytest -v` → **35 passed** ✓ (13 new tests covering caption isolation, heading isolation, paragraph packing, oversized sentence split, cross-paragraph overlap).
- `uv run python scripts/demo_pipeline.py`:
  - PDF: 15 pages → **484 blocks (9 captions) → 50 chunks** (41 paragraph + 9 caption). Compared with the old 1–3 chunks/page, retrieval granularity is dramatically improved.
  - PPTX: → **421 blocks (33 headings) → 68 chunks** (35 paragraph + 33 heading).
  - Manual spot-check on page 3, chunk #1: the previously-merged `Figure 1: The Transformer ...` caption is cleanly separated; section boundaries (`3.1 Encoder...` / `Encoder: ...`) are preserved as `\n\n`; chunk #2 begins with the last sentence of chunk #1 — sentence-level overlap behaves correctly.

### Key Decisions

| Decision | Rationale |
|---|---|
| Captions never share a chunk with body text | Allows retrieval to hit figures/tables directly and prevents off-topic caption text from polluting body chunks. |
| Sentence-level overlap (replaces word-level) | Preserves complete sentences; avoids cutting mid-sentence at chunk boundaries. |
| `max_words=220` (down from 400) | 3–5 chunks per page on the transformer paper — appropriate granularity for both BM25 and embedding retrieval. |
| Write `block_type` into `Chunk.metadata` | Enables downstream weighting/filtering (e.g. "captions only"). |
| `Block` carries `bbox` | Future layout-aware filtering (e.g. drop headers/footers) can use coordinates directly. |

## Stage 2 — Index Construction

**Date:** 2026-05-07

### Process

1. **Retrieval strategy.** Per `CLAUDE.md § 3`, hybrid (BM25 + Voyage AI embeddings) with silent fallback to BM25-only.

2. **`scripts/build_index.py` pipeline:**
   - Load both source files (PDF + PPTX)
   - parse → clean → `chunk_blocks`
   - Write each chunk plus metadata (`chunk_id`, `doc_id`, `page_or_slide`, `block_type`) to JSONL
   - Build the BM25 index over chunk texts; pickle for reuse
   - If `VOYAGE_API_KEY` is set, embed each chunk and persist the vectors

3. **`retrieval.py` — `KnowledgeBase` class:**
   - `load()` — reads JSONL + BM25 pickle; optionally loads Voyage embeddings
   - `search(query, top_k)` — hybrid scoring (BM25 + cosine similarity) when embeddings are available; falls back to pure BM25 otherwise
   - `get_chunk(chunk_id)` / `list_documents()` — read-side accessors

### Key Decisions

| Decision | Rationale |
|---|---|
| JSONL for chunks | Lightweight, streamable line-by-line, easy to debug. |
| Pickle the BM25 model | Avoids rebuilding the O(n) index on every server start. |
| Voyage embeddings with silent fallback | Strong semantic recall to complement BM25 term-frequency; degrades gracefully without an API key. |

### Verification

- `uv run python scripts/build_index.py` completes ✓
- `data/processed/` now contains `chunks.jsonl` + two pickle files ✓
- Sample query `"attention mechanism"` returns high-score chunks (PDF p7, PPTX slide 6) ✓

## Stage 3 — MCP Server

**Date:** 2026-05-07

### Process

1. **Requirements.** Expose three MCP tools — `search`, `get_chunk`, `list_documents` — over Streamable HTTP transport.

2. **FastMCP server (`apps/mcp_server/server.py`):**
   - `@asynccontextmanager lifespan` — `kb.load()` runs once at startup (singleton; avoids reloading on every request).
   - Three `@mcp.tool`-decorated functions:
     - `search(query, top_k=5)` — hybrid retrieval, top-k results
     - `get_chunk(chunk_id)` — single chunk lookup
     - `list_documents()` — indexed documents listing
   - `mcp.run(transport="streamable-http")`.

3. **Local test client (`tests/mcp_client.py`)** — five assertions:
   - `list_tools` — three tools registered ✓
   - `list_documents` — 2 docs, 118 chunks ✓
   - `search("attention mechanism")` — relevant chunks ✓
   - `get_chunk` — single chunk by ID ✓
   - `get_chunk` with unknown ID — returns `null` ✓

4. **Docker image:**
   - `python:3.11-slim` base
   - `uv pip install` for fast dependency install
   - Copies `chunks.jsonl`, BM25 pickles, and `embeddings.npy` into the image
   - Entrypoint: `python apps/mcp_server/server.py`

### Key Decisions

| Decision | Rationale |
|---|---|
| Streamable HTTP (vs stdio) | Production-grade transport; supports concurrent clients, health checks, remote connections. |
| Singleton index via `lifespan` | Eliminates per-request load overhead; native FastMCP pattern. |
| `HOST`/`PORT` from env | Render injects `PORT` automatically; no code changes needed for deploy. |

### Verification

- `uv run python apps/mcp_server/server.py` — boots locally ✓

## Stage 4 — Claude Code Skills

**Date:** 2026-05-07

### Process

1. **Requirement.** Wrap each stage of the preprocessing pipeline as a reusable Claude Code Skill.

2. **Common skill layout:**
   ```
   skills/{skill-name}/
   ├── SKILL.md           # Skill definition and I/O schema
   ├── scripts/run.py     # Entrypoint executable
   └── packages/          # Python module dependencies
   ```

3. **Four skills:**
   - `parse-pdf` — reads a PDF, outputs Block JSON (`text` / `block_type`).
   - `parse-pptx` — reads a PPTX, outputs Block JSON.
   - `clean-text` — takes raw text, returns paragraph-aware cleaned text.
   - `chunk-content` — takes Block JSON (or plain text), returns RAG-ready Chunk JSON.

4. **Safety boundary.** Every skill validates `MAX_FILE_SIZE_BYTES` before processing to prevent runaway resource usage.

5. **Install + invocation.** Skills are installed via `cp -r skills/<name> ~/.claude/skills/` and invoked by natural language inside Claude Code (e.g. *"Parse data/raw/transformer.pdf"*). Outputs are auto-saved to `data/processed/`, with the saved path printed on completion.

### Key Decisions

| Decision | Rationale |
|---|---|
| JSON I/O | Structured; trivial for downstream consumers. |
| Shared `Block` schema across parse + chunk | Single source of truth; seamless pipeline composition. |
| Per-skill file-size cap | Safe execution boundary required by the assignment. |

### Verification

- `uv run python skills/parse-pdf/scripts/run.py --input data/raw/transformer.pdf` ✓
- `uv run python skills/clean-text/scripts/run.py --input <text>` ✓

2. **Decompose** — break the stage into verifiable sub-steps; surface the plan before implementing.
3. **Iterate with verification** — every step has a checkable output (`pytest`, demo log, client response), so regressions surface immediately.
4. **Persist context** — this document is the audit trail; each stage records *what was decided* and *why*.
